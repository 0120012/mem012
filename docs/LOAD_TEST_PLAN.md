# mem012 压测计划

## 目标

对比以下两个生产入口在相同负载下的吞吐、延迟和错误率：

- 4C：`https://mem012.0120012.xyz`
- 1C1G：`https://memory.0120012.xyz`

压测必须从同一台独立客户端顺序执行。两个服务应使用数据量一致的 `loadtest` profile；认证密钥只通过环境变量传入。

## 客户端准备

在独立压测客户端执行：

```sh
python3 -m pip install --requirement scripts/requirements.txt
```

## 测试分层

1. `GET /`：验证 HTTPS 与静态文件服务。
2. `GET /api/health`：验证 OpenResty 反代和 Rust HTTP 基础开销，不代表数据库性能。
3. `GET /api/memories`：携带 `mem_session` Cookie 与 `X-Mem-Project: loadtest`，验证真实数据库读取。

本轮只压只读接口。写入、图重建和删除接口必须使用可丢弃数据，并另行制定计划。

## 负载阶梯

每个目标先执行 5 次单请求预检，再按下表逐级运行；两台服务器在同一级别使用完全相同的参数。

| 阶段 | 并发 | 持续时间 |
|---|---:|---:|
| 预热 | 5 | 30 秒 |
| 低负载 | 10 | 60 秒 |
| 常规负载 | 25 | 120 秒 |
| 中负载 | 50 | 120 秒 |
| 高负载 | 100 | 120 秒 |

1C1G 服务器只有在 50 并发仍满足停止条件时才进入 100 并发。4C 服务器完成相同阶梯后，才能按结果决定是否增加 150、200 并发。

## 指标

每个阶段记录：请求数、成功数、状态码分布、RPS、平均延迟、p50、p95、p99、最大延迟和错误样本。服务器侧同步记录 CPU、内存、swap、PostgreSQL 容器资源和服务日志。

初始验收线：

- 失败率 `< 1%`
- p95 `< 500 ms`
- p99 `< 1000 ms`
- 无 OOM、进程重启、连接池超时或持续内存增长

## 自动停止条件

任一级别满足以下任一条件时，立即停止该目标后续升压：

- 最近阶段失败率 `>= 5%`
- p99 `>= 3000 ms`
- 连续出现 HTTP 5xx 或连接失败
- 服务进程重启、数据库不可用或服务器发生 OOM

## 执行顺序

先测试 1C1G，再等待 3 分钟冷却并确认服务恢复；随后测试 4C。每个目标按 `/`、`/api/health`、`/api/memories` 的顺序执行。最终只比较相同接口、相同并发和相同数据规模下的结果。
