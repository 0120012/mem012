# mem012 千人规模 PostgreSQL 连接优化计划

## 目标

面向约 1000 名注册用户，建立少量、长期复用、可以排队和快速失败的 PostgreSQL 连接体系。

1000 名用户不等于需要 1000 条数据库连接。系统应按实际同时执行的数据库操作数配置连接，而不是按注册用户数配置连接。

## 当前证据

当前工具通过独立 CLI 进程执行。每次调用都会启动进程、读取配置、创建当前 profile 的连接池、连接 PostgreSQL、执行查询，然后随进程退出销毁连接池。

只读 `search_memory` 并发实测结果：

| 并发 | 成功率 | 吞吐量 | p50 | p95 | p99 |
|---:|---:|---:|---:|---:|---:|
| 20 | 100% | 8.26/s | 2.262s | 2.390s | 2.455s |
| 50 | 100% | 12.51/s | 3.080s | 3.970s | 4.917s |
| 100 | 100% | 13.86/s | 3.571s | 6.038s | 6.849s |
| 200 | 100% | 14.99/s | 7.888s | 11.715s | 12.747s |

从 100 增加到 200 并发时，吞吐量只增加约 8%，p95 却接近翻倍。当前瓶颈表现为连接建立或资源争用，而不是缺少更多并发请求。

## 目标架构

```text
用户或 Agent
    ↓
OpenResty
    ↓
mem012 常驻服务
    ├─ 共享 ServerState
    ├─ 共享、受限的 profile 连接池
    └─ 连接池等待与超时控制
    ↓
PgBouncer（验证后引入）
    ↓
PostgreSQL
```

CLI 后续只作为客户端，不再为每次工具调用独立持有数据库连接池：

```text
mem012 CLI → 本机常驻 mem012 服务 → 共享 PgPool → PostgreSQL
```

## 第一阶段：应用内共享连接池

### HTTP 服务

`mem012 server` 启动后持有共享连接池，请求只从池中借用连接。禁止每个请求创建或销毁 `PgPool`。

建议初始参数：

```text
min_connections = 0
acquire_timeout = 3 秒
idle_timeout = 5 分钟
max_lifetime = 30 分钟
```

连接池上限按部署规格保守设置：

| 服务器 | 实例数 | 每实例 `max_connections` |
|---|---:|---:|
| 1C1G | 1 | 4～6 |
| 2C4G | 1 | 8～12 |
| 2C4G | 2 | 每实例 4～6 |

连接池不是越大越快。过多同时执行的 SQL 会增加 PostgreSQL 上下文切换、内存消耗和磁盘争用。

### 工具调用

工具调用应逐步进入常驻服务，复用服务持有的配置和连接池。CLI 保留现有调用外观，但内部转发到本机服务。

在完成该改造前，不通过扩大 PostgreSQL `max_connections` 掩盖每次 CLI 调用重新建连的问题。

## 第二阶段：限制 profile 连接池总量

如果 1000 名用户对应大量独立 profile，则不能为每个 profile 永久保留完整连接池。例如每个 profile 上限为 4 时，1000 个 profile 理论上可产生 4000 条连接。

独立 profile 数据库模式下采用：

```text
profile 连接池惰性创建
每个 profile 最多 1 条物理 PostgreSQL 连接
全局最多 20 条 profile 物理连接（包含空闲连接）
空闲 5 分钟后关闭连接
达到预算时先关闭最久未使用且无活跃连接的池，再创建新连接
没有可回收连接时排队或快速失败
```

全局许可在物理连接创建时取得，并持续占用到该连接关闭；不能在事务结束时释放。这样全局预算覆盖活跃和空闲连接，而不仅是正在执行的事务。连接池元数据数量可以高于 20，但不得保留额外的物理 PostgreSQL 连接。

长期需要评估是否将 profile 改为共享数据库和共享池，通过 `profile_id` 隔离数据。该变化涉及数据模型与隔离边界，不与本轮连接复用改造混合实施。

## 第三阶段：仅在固定数据库集合后引入 PgBouncer

每个 profile 使用独立 database 和 role 时，PgBouncer 会为每个 `(database, user)` 保留独立后端池；即使应用关闭客户端连接，PgBouncer 的空闲后端仍会保留到 `server_idle_timeout`。因此当前拓扑不能用 PgBouncer 承接 20 条全局后端连接目标，第二阶段必须保持应用直接管理物理连接。

只有迁移到共享数据库，或把数据库集合固定为可枚举且有限的数量后，才评估 PgBouncer 事务池模式。此时 `default_pool_size` 仍是每个 `(database, user)` 的上限，所有池大小之和必须不超过 20；`max_client_conn` 只限制客户端连接数，不能替代该总和约束。

PostgreSQL 总连接预算固定为 25 条，其中 20 条给 profile 工作连接，5 条按部署版本的运维保留机制配置给恢复和管理连接。这是跨所有 database、role 和连接池的最终硬上限。

启用事务池前必须确认代码不依赖：

- session 级临时表；
- 跨事务的 session `SET`；
- 跨事务 prepared statement 状态；
- 长时间持有的事务或连接。

## 超时与过载保护

建议初始边界：

```text
连接池获取超时 = 3 秒
普通 SQL 执行超时 = 5～10 秒
写事务超时 = 10～15 秒
锁等待超时 = 3 秒
空闲事务超时 = 10 秒
```

连接池耗尽时应快速返回明确的繁忙错误或 HTTP 503，不能无限等待并持续堆积请求。

PostgreSQL 侧需要结合现有查询验证以下设置：

```sql
statement_timeout = '10s'
idle_in_transaction_session_timeout = '10s'
lock_timeout = '3s'
```

## 容量假设与初始目标

初始容量模型：

```text
注册用户约 1000
日活用户约 100～200
同时在线约 20～50
同时执行数据库操作约 5～20
```

近期目标：

```text
稳定工具并发 = 20～50
PostgreSQL 真实工作连接 = 8～20
工具吞吐 = 30～50 次/秒
p95 < 3 秒
错误率 < 1%
```

这些数值是待验证目标，不是当前已经达到的容量承诺。最终容量必须以生产规格服务器、真实数据量和完整工具调用链的持续压测为准。

## 监控指标

优化前后至少记录：

- 请求量、成功率和超时率；
- p50、p95、p99；
- 连接池活跃连接、空闲连接和等待队列；
- 连接池获取耗时与获取超时次数；
- PostgreSQL 当前连接数、活跃查询数和事务数；
- 慢查询、锁等待和死锁；
- 各 profile 的池数量与最后使用时间；
- 应用和 PostgreSQL 的 CPU、内存及磁盘等待。

## 实施顺序

每项作为独立、可验证的最小切片实施：

1. 为 HTTP API 建立共享连接池，消除每请求建池。
2. 增加连接池获取超时和基础监控。
3. 让一个只读工具通过常驻服务复用连接池。
4. 使用相同参数复测该工具，确认进程启动和建连开销下降。
5. 逐步迁移其余只读工具，再迁移事务写入工具。
6. 增加 profile 池总量上限和空闲淘汰。
7. 在应用池稳定后验证 PgBouncer 事务模式。
8. 根据慢查询证据决定是否调整 SQL、索引、分页或数据模型。

## 验收标准

- HTTP 请求和工具调用均不再为每次请求创建连接池。
- 同一 profile 的请求能够复用连接。
- profile 池和 PostgreSQL 真实连接数均有明确上限。
- 连接池耗尽时能在规定时间内快速失败。
- 错误状态码、错误语义和 JSON 响应契约保持一致。
- 50 并发工具测试错误率低于 1%，p95 低于 3 秒。
- 100 和 200 并发时系统可以受控排队或拒绝，不能内存失控或拖垮数据库。
- 优化后的吞吐、延迟和数据库连接数均有可重复的对比数据。

## 明确不做

- 不直接把 PostgreSQL `max_connections` 提高到数百或上千。
- 不按注册用户数预创建连接。
- 不为所有 profile 永久保留连接池。
- 不在缺少慢查询证据时盲目增加索引。
- 不将分页、数据模型迁移和连接池优化放入同一个开发切片。
- 不把单次峰值压测结果当作长期容量承诺。
