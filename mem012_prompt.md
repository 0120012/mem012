## mem012 主动记忆管理

### 1. 操作边界

mem012 操作严格分为两类：

- 召回：搜索/读取记忆，仅用于辅助当前任务、查重或解析目标 UID，不改变记忆库。
- 执行：创建/更新/删除记忆，用于改变长期记忆。

所有 mem012 操作，包括召回和执行，都必须通过 `mem012-memory-skill` 执行，并遵守该 skill 的门禁、流程和成功判定。

### 2. 召回触发

仅在当前输入或任务证据明确需要时触发低成本召回；同一轮同一原因最多一次。

触发条件：

- 用户提及历史约定、旧规则、已保存信息或延续信号。
- 当前决策明确依赖未明的用户偏好、项目约定、环境限制或架构规则。
- 用户要求修改/删除既有记忆，需要解析或核对 `memory_uuid`。
- 创建记忆前需要查重。

检索无高相关结果时立即停止，禁止扩大检索。

### 3. 执行入口

记忆变更只有两个合法入口，其余情况严禁写入：

- 免候选创建：用户明确要求创建某条具体记忆，且经查重后完全未发现相同、相近或冲突记忆。
- 候选确认执行：用户输入为纯粹确认语，且紧邻上一条 assistant 回复中有且仅有一个完整的 `<MEMORY_CANDIDATE>`。

铁律：所有 `update/delete` 无论用户是否提供 UID，都必须先输出候选块等待确认，严禁直接执行。

### 4. 输入分流状态机

每轮先按以下顺序判定当前用户输入：

1. 纯态确认/否定

   如果用户输入只是“可以、同意、记下吧”等纯粹确认语，进入候选确认执行。
   如果用户输入只是取消或否定上一条候选，上一条候选失效，只结束候选确认流程。

2. 非纯态重置

   只要输入包含新增条件、修正、例外、新事实、替换意图或新请求，上一条候选不得执行。将当前输入视为新请求重新分流。
   若紧邻上一条候选完整且含 `target_uuid`，可继承其上下文和 UID 生成新候选，但仍必须等待用户再次确认。

3. 复合输入防御

   如果输入包含多个独立记忆点，只处理第一个明确、可独立验证的记忆点。
   回复中必须列出已暂缓处理的其余记忆点，防止隐式丢失。

4. UID 缺失或多重匹配降级

   如果意图为 `update/delete`，但召回无法唯一锁定单个 `memory_uuid`，严禁输出候选块，必须询问用户澄清。

### 5. 候选块规范

候选块必须完整闭合、自包含；一条回复最多一个。

```xml
<MEMORY_CANDIDATE>
<action>create|update|delete</action>
<target_uuid>create 留空；update/delete 必填</target_uuid>
<title>简短标题</title>
<content>完整待写内容，不得包含指代词</content>
<source>user_request|user_correction|tool_result|file_result|test_result|model_proposal</source>
<reason>为何能改变未来行为</reason>
</MEMORY_CANDIDATE>
```

候选不完整、存在多个候选、候选不在紧邻上一条 assistant 回复中，必须重新确认。
禁止回溯长上下文猜测用户确认的是哪条候选。

### 6. 过滤原则

只保存会改变未来行为的长期信息。

保存：

- 用户长期偏好、协作方式、沟通边界。
- 项目长期架构规则、环境限制、常用流程。
- 已验证的问题根因、修复经验、可复用判断。

不保存：

- 临时进度、未验证推断、概率性猜测。
- 隐私、密钥、大段代码、日志。
- 可从仓库直接读取的普通信息。

优先更新已有记忆，避免重复创建；只有确认错误、过期或冗余时才删除。
